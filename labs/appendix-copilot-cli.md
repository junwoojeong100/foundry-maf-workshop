# 부록. GitHub Copilot CLI 실습 *(Optional)*

> ⏱️ 예상 시간: 15분 · 시간이 남을 때 진행하는 선택 실습입니다.

터미널에서 자연어로 코딩을 돕는 **GitHub Copilot CLI**를 체험합니다. 앞선 실습이 "AI 에이전트를 **만드는**" 쪽이었다면, 이 부록은 "AI 에이전트를 **쓰는**" 쪽 — 즉 개발자 도구로서의 코딩 에이전트를 다룹니다.

> ⚠️ **진행 전 확인**: Copilot CLI는 빠르게 업데이트됩니다. 아래 명령어(패키지 이름, 실행 명령)는 실습 직전 **공식 문서에서 한 번 검증**하세요.
> - 공식 문서: https://docs.github.com/copilot/concepts/agents/about-copilot-cli
> - 설치 가이드: https://docs.github.com/copilot/how-tos/set-up/install-copilot-cli

---

## 🎯 목표

- Copilot CLI를 설치하고 로그인한다.
- **두 가지 실행 모드**(대화형 / 프로그래매틱)와 **Plan 모드**를 이해한다.
- **권한(승인) 모델**을 이해하고 도구 실행을 통제한다.
- 꼭 알아야 할 **슬래시 명령어**와 **세션 관리**를 익힌다.
- MCP·커스텀 인스트럭션 개념을 본편의 도구(tool) 개념과 연결한다.

---

## 사전 요구사항

- **Node.js 22 이상** (`node --version`으로 확인) 및 npm
- **GitHub 계정** + Copilot 구독 (Copilot Free/Pro/Business 등)

> 💡 Copilot CLI는 앞 랩에서 만든 MAF 에이전트와 **똑같은 골격**을 가진 코딩 에이전트입니다: 자연어 요청 → 모델이 **도구(파일 읽기/쓰기, 셸 명령, GitHub API)** 호출을 판단 → **사용자 승인** → 실행. Lab 04의 `approval_mode`가 여기서는 **권한 모델**로 나타납니다.

## 1. 설치

```bash
npm install -g @github/copilot
```

설치 확인:

```bash
copilot --version
```

> 💡 조직 정책이나 버전에 따라 패키지/명령 이름이 다를 수 있습니다. `copilot` 명령이 없다면 공식 설치 가이드를 확인하세요.

## 2. 실행 & 로그인

```bash
copilot
```

처음 실행하면 GitHub 인증을 안내합니다. 화면의 지시에 따라 브라우저에서 로그인하거나, 대화형 세션 안에서 `/login`을 입력해 인증합니다.

## 3. 두 가지 실행 모드 (꼭 이해하기)

Copilot CLI는 크게 두 가지 방식으로 씁니다.

| 모드 | 실행 | 언제 쓰나 |
|------|------|-----------|
| **대화형(Interactive)** | `copilot` | 사람이 앉아서 주고받으며 작업. 기본 모드 |
| **프로그래매틱(Programmatic)** | `copilot -p "..."` | 스크립트·CI 등 자동화에서 한 번에 실행 (비대화형) |

### 3-1. 대화형 모드

```bash
copilot
```

대화형 프롬프트가 열리면 자연어로 요청합니다.

```
현재 폴더에 있는 파이썬 파일들이 무슨 일을 하는지 설명해줘
```

대화형에는 **세 가지 모드**가 있고, **Shift+Tab**으로 순환 전환합니다(표준 → Plan → Autopilot).

| 모드 | 동작 |
|------|------|
| **표준(Standard)** (기본) | 요청을 바로 수행 — 필요한 도구를 호출하고 승인을 받아 실행 |
| **Plan 모드** | 코드를 쓰기 **전에** 요청을 분석하고, 필요한 질문을 던져 범위를 좁힌 뒤 **계획을 먼저 제시**. 큰 작업 전 방향을 맞출 때 유용 |
| **Autopilot 모드** | 승인 프롬프트 없이 도구를 자동 실행 — 빠르지만 통제력이 낮으므로 신뢰할 수 있는 작업에만 |

### 3-2. 프로그래매틱 모드

한 번의 프롬프트로 끝내고 결과를 받습니다. 자동화·파이프라인에 적합합니다.

```bash
# 결과를 표준출력으로 받아 파이프에 넘길 수 있음
copilot -p "이 저장소의 README를 3줄로 요약해줘"
```

> ⚠️ 프로그래매틱 모드는 대화가 없으므로, 도구 실행 승인을 그때그때 받을 수 없습니다. 필요한 도구는 아래 **권한 플래그**(`--allow-tool` 등)로 미리 허용해야 합니다.

---

## 4. 권한(승인) 모델 — 안전의 핵심

Copilot CLI는 **파일 수정·셸 명령 실행·네트워크 호출** 같은 민감한 동작 전에 **사용자 승인**을 요구합니다. 이것이 코딩 에이전트를 안전하게 쓰는 핵심 장치입니다. (Lab 04의 `approval_mode="always_require"`와 같은 원리)

승인 프롬프트가 뜨면 보통 이렇게 선택합니다.

- **이번만 허용** — 그 명령 한 번만 실행
- **세션 동안 이 도구 허용** — 같은 도구(예: `git`)는 이후 묻지 않음
- **거부** — 실행을 막고 다른 방식을 유도

명령줄 플래그로 승인 정책을 미리 정할 수도 있습니다.

| 플래그 | 의미 |
|--------|------|
| `--allow-tool <도구>` | 특정 도구를 프롬프트 없이 허용 (예: `--allow-tool write`, `--allow-tool 'shell(git:*)'`) |
| `--deny-tool <도구>` | 특정 도구를 항상 금지 (예: `--deny-tool 'shell(rm)'`) |
| `--allow-all-tools` | **모든 도구**를 프롬프트 없이 허용 — 비대화형(`-p`) 실행에 필요 |
| `--allow-all` (별칭 `--yolo`) | **모든 권한**(도구·파일 경로·URL)을 허용 — 편하지만 위험 |

> ⚠️ `--allow-all` / `--yolo`는 확인 없이 파일 삭제·명령 실행까지 다 해버릴 수 있습니다. 신뢰할 수 있는 작업, 격리된 환경에서만 쓰세요. 실습에서는 **기본 승인 흐름을 그대로 두는 것**을 권장합니다.

## 5. 꼭 아는 슬래시 명령 (대화형 세션 내)

세션 안에서 `/`로 시작하는 명령을 씁니다. 자주 쓰는 것들:

| 명령 | 설명 |
|------|------|
| `/help` (또는 `?`) | 사용 가능한 명령 목록 |
| `/login` · `/logout` | GitHub 인증 / 로그아웃 |
| `/model` | 사용할 모델 선택·전환 |
| `/context` | 현재 **토큰 사용량**을 시각적으로 확인 |
| `/compact` | 대화 기록을 수동으로 **압축**(요약)해 컨텍스트 확보 |
| `/add-dir <경로>` | 신뢰할 작업 디렉토리를 추가 |
| `/cwd` (또는 `/cd`) | 현재 작업 디렉토리 전환 |
| `/agent` | 사용 가능한 **커스텀 에이전트** 선택 |
| `/mcp add` | **MCP 서버** 추가 (아래 6절 참고) |
| `/resume` | 저장된 세션을 골라 이어가기 |
| `/usage` | 세션 통계(토큰·비용 등) 보기 |
| `/exit` (또는 `/quit`) | 세션 종료 |

> 지원 명령은 버전에 따라 달라집니다. 세션에서 `/help`로 최신 목록을 확인하세요.

---

## 6. 세션 관리 & 컨텍스트

대화가 길어지면 토큰이 한도에 가까워질 때 **자동 압축(compaction)** 이 일어나 이전 내용을 요약합니다. 수동으로는 `/compact`로 압축하고, `/context`로 현재 사용량을 확인합니다.

세션을 나갔다가 이어서 하려면 CLI를 이렇게 실행합니다.

```bash
copilot --continue     # 가장 최근에 닫은 로컬 세션을 바로 이어감
copilot --resume       # 저장된 세션 목록에서 골라 이어감
```

> 💡 **Copilot Memory**: Copilot은 프로젝트에서 파악한 코딩 관례·패턴을 기억해, 세션이 바뀌어도 매번 설명하지 않아도 되게 합니다.

---

## 7. 확장: MCP 서버 & 커스텀 인스트럭션 (개념)

- **MCP(Model Context Protocol) 서버**: Copilot에 **새 도구·데이터 소스**를 붙이는 표준 방식입니다. `/mcp add`로 서버를 등록하면(값 입력 후 `Ctrl+S` 저장), 그 서버가 제공하는 도구를 Copilot이 호출할 수 있습니다. → 본편에서 MAF 에이전트에 `@tool`을 붙인 것과 같은 "도구 확장" 개념입니다.
- **커스텀 인스트럭션**: 프로젝트별 지침 파일로 Copilot의 작업·테스트 방식을 안내합니다. → MAF `Agent`의 `instructions`와 같은 역할입니다.

## 🔬 실습 미션 (자유롭게)

1. **코드 설명**: 앞 랩에서 만든 `code/maf_tools/main.py`를 열고, Copilot에게 "이 코드가 하는 일을 초보자에게 설명해줘"라고 요청.
2. **Plan 모드 체험**: `Shift+Tab`으로 Plan 모드로 바꾼 뒤 "이 저장소에 간단한 테스트를 추가해줘"라고 요청 → 바로 코드를 쓰지 않고 **계획부터 제시**하는 흐름을 관찰.
3. **코드 생성**: "섭씨를 화씨로 바꾸는 파이썬 함수를 테스트와 함께 만들어줘" → 파일 생성 **승인 프롬프트**가 뜨는지 확인.
4. **슬래시 명령**: `/context`로 토큰 사용량을 보고, `/compact`로 대화를 압축해보기.
5. **프로그래매틱 모드**: 세션을 나와 `copilot -p "이 폴더에서 가장 최근에 수정된 파일 5개를 보여줘"`를 실행 → 대화형과의 차이를 비교.

각 단계에서 Copilot이 **무엇을 하려는지 먼저 확인**하고 승인하는 흐름에 주목하세요. 이것이 안전한 AI 에이전트 사용의 핵심입니다.

## ⚠️ 트러블슈팅

| 증상 | 해결 |
|------|------|
| `command not found: copilot` | 설치 확인(`npm ls -g @github/copilot`), npm 전역 bin 경로가 PATH에 있는지 확인 |
